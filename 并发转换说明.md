# 并发转换说明

## 当前行为

程序的批量转换并发数由设置中的“最大并发数”控制，实际会按下面规则生效：

- 默认值为 `4`
- 上限为当前机器的 CPU 核心数
- 实际启动值为 `min(设置的最大并发数, 本次选中的文件数)`

例如：

- 选中 `8` 个文件，设置并发数为 `4`，会同时启动最多 `4` 个转换任务
- 选中 `2` 个文件，设置并发数为 `8`，实际只会同时启动 `2` 个转换任务

## 这次检查的结论

代码里没有把转换线程硬编码限制为 `3`。本次检查时确认：

- 设置值会从界面正确传到 `ConversionWorker`
- `ConversionWorker` 会把该值原样传给 `ThreadPoolExecutor(max_workers=...)`
- 本地复现中，配置 `4` 个并发时，批量转换可以稳定同时运行 `4` 个任务

## 为什么之前看起来线程数不对

本次顺手修复了两个会让观感失真的问题：

- 进度对话框里的“正在转换文件”列表之前使用文件名作为唯一键  
  如果同时转换的文件存在重名，不同任务会被合并显示
- 活动线程数之前在任务交接的短时间窗口里可能统计偏大或偏小  
  这会让界面显示和真实运行状态短暂不一致

## 本次修复

- 活动任务改为按完整文件路径识别
- 当存在重名文件时，会自动补充父目录名区分显示
- 鼠标悬停时仍可看到完整路径
- 活动线程数改为在任务真正结束时回收统计，避免交接时的瞬时误差

因此，当前版本中：

- 线程池的实际并发数按设置生效
- 进度对话框也能更准确地反映正在运行的多个任务
